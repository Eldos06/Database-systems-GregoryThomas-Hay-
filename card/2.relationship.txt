a. How We Ended Up with Inefficient Hierarchical Systems | After World War II, early computers were designed to copy the way people organized information on paper. Businesses used paper folders and filing cabinets for thousands of years, so the first hierarchical databases were built to look the same ‚Äî with one ‚Äúmain folder‚Äù at the top and all other records arranged under it. At first this looked logical, but soon it became a problem. Every time a company stored data, it had to duplicate the same values in many hierarchies. When someone changed one value, they had to update all copies. This made INSERT, UPDATE, and DELETE operations very slow because each action had to be repeated in every hierarchy. Another issue was data integrity: people sometimes kept ‚Äúsecret‚Äù copies to work faster, causing errors and conflicts. Each query required reading almost every record even if only 1 % was useful ‚Äî a huge waste of memory in the 1960s. b. What Is Meant by ‚ÄúPaper-Based Systems‚Äù | Before electronic databases, even large companies used paper bookkeeping. Employees recorded everything by hand ‚Äî invoices, attendance, orders ‚Äî on paper forms. Communication happened by letters, telephone, or typewriter. These ‚Äúpaper-based systems‚Äù were the ancestors of hierarchical databases: each document had one main topic (like a customer file) with many sub-pages. They were rigid and slow; information in one folder was not linked to another. c. What Are Data Types and When to Use Them | In relational databases, a data type defines what kind of values can be stored in a column. It ensures accuracy and prevents errors ‚Äî for example, dates in DATE fields or numbers in INT fields. Correct data types improve speed and security. Main categories: Numeric (INT, DECIMAL, FLOAT), Date/Time (DATE, DATETIME2), String (VARCHAR, NVARCHAR), Money (MONEY), Binary (VARBINARY). Choosing the right type saves space, makes queries faster, and keeps data clean. 	 c. –ß—Ç–æ —Ç–∞–∫–æ–µ —Ç–∏–ø—ã –¥–∞–Ω–Ω—ã—Ö –∏ –∫–æ–≥–¥–∞ –∏—Ö –∏—Å–ø–æ–ª—å–∑–æ–≤–∞—Ç—å | –í —Ä–µ–ª—è—Ü–∏–æ–Ω–Ω—ã—Ö –±–∞–∑–∞—Ö –¥–∞–Ω–Ω—ã—Ö —Ç–∏–ø –¥–∞–Ω–Ω—ã—Ö –æ–ø—Ä–µ–¥–µ–ª—è–µ—Ç, –∫–∞–∫–∏–µ –∑–Ω–∞—á–µ–Ω–∏—è –º–æ–∂–Ω–æ —Å–æ—Ö—Ä–∞–Ω—è—Ç—å –≤ —Å—Ç–æ–ª–±—Ü–µ. –û–Ω –æ–±–µ—Å–ø–µ—á–∏–≤–∞–µ—Ç —Ç–æ—á–Ω–æ—Å—Ç—å –∏ –ø—Ä–µ–¥–æ—Ç–≤—Ä–∞—â–∞–µ—Ç –æ—à–∏–±–∫–∏ ‚Äî –Ω–∞–ø—Ä–∏–º–µ—Ä, –¥–∞—Ç–∞ –≤ –ø–æ–ª–µ DATE –∏–ª–∏ —á–∏—Å–ª–æ –≤ INT. –ü—Ä–∞–≤–∏–ª—å–Ω—ã–π —Ç–∏–ø –¥–∞–Ω–Ω—ã—Ö —É—Å–∫–æ—Ä—è–µ—Ç —Ä–∞–±–æ—Ç—É –∏ –ø–æ–≤—ã—à–∞–µ—Ç –±–µ–∑–æ–ø–∞—Å–Ω–æ—Å—Ç—å. –û—Å–Ω–æ–≤–Ω—ã–µ –∫–∞—Ç–µ–≥–æ—Ä–∏–∏: —á–∏—Å–ª–æ–≤—ã–µ (INT, DECIMAL, FLOAT), –¥–∞—Ç–∞/–≤—Ä–µ–º—è (DATE, DATETIME2), —Å—Ç—Ä–æ–∫–æ–≤—ã–µ (VARCHAR, NVARCHAR), –¥–µ–Ω–µ–∂–Ω—ã–µ (MONEY), –±–∏–Ω–∞—Ä–Ω—ã–µ (VARBINARY). –í—ã–±–æ—Ä –ø—Ä–∞–≤–∏–ª—å–Ω–æ–≥–æ —Ç–∏–ø–∞ —ç–∫–æ–Ω–æ–º–∏—Ç –ø–∞–º—è—Ç—å, —É—Å–∫–æ—Ä—è–µ—Ç –∑–∞–ø—Ä–æ—Å—ã –∏ —Å–æ—Ö—Ä–∞–Ω—è–µ—Ç —á–∏—Å—Ç–æ—Ç—É –¥–∞–Ω–Ω—ã—Ö.
d. The ‚ÄúBrilliant Breakthrough‚Äù of Relational Theory and Its Longevity | In 1969 Dr. E. F. Codd from IBM proposed the relational model. Instead of many hierarchies, data is stored once in tables linked by primary and foreign keys. This removed duplication and allowed powerful queries without changing storage. Users could analyze data freely through SQL. Because the model is logical and mathematical, it has remained stable for almost 60 years and is used by Oracle, PostgreSQL, and SQL Server.
